01 / End-to-end flow

Every gate.
In order.

The orchestrator is a state machine with judgment. It can parallelize work inside a phase, but it cannot skip the product contract or evidence gates.

P0Intake

Extract users, problem, measurable outcome, constraints, and acceptance scenarios. Ask only questions that materially alter the build.

GATE: executable PRD
P1Simplify

Find the smallest architecture and product surface capable of hitting the KPI. Record tempting work that is deliberately excluded.

GATE: no overbuild
P2Specify

Create the technical spec, UX flow, implementation phases, test plan, ownership boundaries, and rollback strategy.

GATE: buildable plan
P3Parallel build

Create one worktree per independent task. Assign workers bounded outputs and file ownership. Keep the main branch clean.

GATE: task evidence
P4Review loop

Run tests, static checks, code review, UI computer-use QA, and KPI checks. Feed concrete failures back to the owner.

GATE: reviewer pass
P5Integrate

Compare candidates, merge the best verified work, resolve interactions, and rerun the whole-system test suite.

GATE: main is green
P6Release

Produce release evidence, known limitations, runbook, and a clear ship/no-ship decision tied to the original KPI.
