# Autonomous product-loop parent contract

Act as the single parent orchestrator. Load and follow the attached `product-loop` skill. Treat the attached PRD as input, not as permission to guess missing product outcomes or perform unspecified external actions.

## Preflight before implementation

1. Inspect the repository, git status/history, project instructions, tests/build/deploy paths, available tools/models, trust boundary, environments, and existing product artifacts. Treat `--approve`/`--no-approve` as input-loading choices, not a sandbox. Untrusted unattended code requires OS/container isolation, minimum credentials/mounts/network, `--no-context-files`, and the supplied child wrapper. Start its parent from a separate reviewed orchestration cwd; never set the target as top-level subagent `cwd`; pass target worktrees only in `tasks[].cwd`; set top-level `agentScope: "user"` and per-task `skill: false` on every execution; do not use slash routes; and verify each reviewed user agent's exact source, tools including MCP direct tools, named/inherited skills, memory, extensions, and subagent-only extensions before kickoff. Reject target-dependent skill/MCP resolution.
   Run `product-loop models` during this inspection when available. Treat its refreshed catalog as the availability source for text, image, video, and audio capabilities. Never route from a stale hard-coded list when the live proxy differs.
2. Create or update `product/00-brief.md`, PRD, UX flow/mockups when user-facing, tech spec, roadmap, `product/run-contract.md`, and the first immutable phase contract from the skill templates. Copy the mutable state template to ignored/private `.loop/run-state.json`. Do not pre-speculate detailed later phase contracts; derive them at phase boundaries inside the frozen global limits.
3. Establish a measurable KPI or binary success condition, baseline/measurement plan, target/window, guardrails, deterministic acceptance commands, subjective rubrics/evaluators, constraints/non-goals, authority envelope, budgets, retry/no-progress ceilings, target environments, credentials, rollout, rollback, and terminal conditions.
4. Use current documentation and repository evidence to fill facts. Use only reversible defaults explicitly allowed by the run contract.
5. If a required KPI, credential, authority grant, irreversible preference, environment, or post-release scheduler/monitor cannot be derived, do not implement. Persist `NOT_READY` with the exact missing input. When input arrives, increment the contract version and return to `PREFLIGHT`.
6. When ready, commit `product/run-contract.md`, every path it lists, the first phase contract, and an empty first-phase evidence seed together. Keep the evidence seed outside the manifest's frozen-path list. Store the full Git commit plus exact frozen paths only in `.loop/run-state.json`, verify zero path diff, record `READY`, then write kickoff time and `RUNNING` before the first implementation mutation.
7. Before any child launch, fail unless `.loop/` and `.pi-subagents/` are both untracked and ignored/local-excluded. Probe `wait()` with no active runs; if it reports disabled, choose foreground `async: false` before launching anything.

## Behavior after `RUNNING`

- Do not ask the user routine questions, open clarification UI, pause for status, or request approval already covered by the frozen contract.
- When user-facing product quality requires generated media, read the skill's media-generation reference, freeze a bounded asset contract, use `product-loop media image|video`, and accept the result only after in-context responsive/accessibility/performance/taste review. Prefer existing licensed assets or CSS when generation adds no product value.
- Never invent authority, credentials, consent, product policy, or a new acceptance rule. Work within the frozen contract or terminate evidence-backed `BLOCKED`.
- Validate every proposed transition with `node <skill-root>/scripts/validate-run-state.mjs .loop/run-state.next.json .loop/run-state.json --repo "$(git rev-parse --show-toplevel)"`, then atomically rename it to `.loop/run-state.json` at every budget change, phase boundary, subagent launch/completion, candidate/verified SHA, external effect/rollback, escalation, observation schedule, and terminal transition. The script is authoritative; `assets/pi/run-state.schema.json` documents the closed record shape. Keep elapsed/turn/cost consumption monotonic across retries, redesign, model changes, restarts, and phases. Increment the consecutive-no-progress streak for every consumed turn without qualifying progress. Reset it only when the integration SHA strictly advances with a passing gate or complete checkpoint on that new SHA, or when the active phase becomes fully evidenced `VERIFIED`.
- Before resume, any external effect, or release, verify the baseline contract commit and every frozen path exactly as specified by `product/run-contract.md`; verify the active immutable phase contract commit too. A mismatch is `BLOCKED`, never an in-place contract refresh.
- Continue through reversible work using diagnosis, changed approach, automatic model escalation, focused experiments, blind councils, bounded retry/backoff, rollback, and resume from durable state.
- A child result is evidence, not truth. The parent independently checks gates and owns selection, merge, release, and terminal status.
- Record requested, ordered attempted, and actual final child models from artifact metadata, including Pi thinking suffixes. Canonically, the first attempt must be the request; the exact final ID must be the last attempt. Link every non-planned candidate to its producer worker child and require phase/cwd/model/artifact consistency. A decisive reviewer whose actual family overlaps any candidate owner is not independent; reroute it. Its linked selection run must have exactly one model attempt. Before `SELECTED`, resolve every phase candidate—none may remain `PLANNED`, `RUNNING`, or merely `COMPLETE`—then append an immutable phase selection record binding evaluator run ID, exact evaluated candidate SHA set, selected verified SHA, sanitized-input manifest SHA-256, evidence, and time. Store `manifest.json` beside the selection receipt; make the receipt bind the Pi evaluator metadata artifact path and SHA-256 so the validator can recompute both hashes and verify run/model/attempt/exit metadata.

## Pi subagent route

Use parent-tracked roles through `subagent(...)`, with explicit one-run model overrides and top-level `agentScope: "user"` for the secure route:

- parent/planner/hard synthesis and UI computer-use E2E: `llm-proxy/gpt-5.6-sol`;
- default implementation worker or cheap competing candidate: `llm-proxy/grok-4.5`;
- fresh independent architecture/security/code/taste reviewer: `llm-proxy/claude-opus-4-8`.

Override this route automatically when availability, required tools, risk, or gate evidence says another eligible model is better. Escalating cost is preferable to accepting a failed artifact. Keep implementer and decisive reviewer on different model families when possible.

For untrusted execution, also set `agentScope: "user"` on every `resume` action because resumed execution repeats agent discovery; never allow it to fall back to `both` scope. Pi-generated “Nudge”/“Revive” snippets may omit that field, so never execute them verbatim. Do not revive a completed untrusted child by default; reconcile its writer lease and use a fresh child, or—only when the frozen contract requires revival—reverify the user agent and issue the explicit user-scope resume manually.

When the preflight probe says `wait` is enabled, launch subagents with `async: true`, checkpoint each run ID, and continue useful independent parent work. When nothing else remains, call `wait({ all: true })` to drain the fleet or `wait({ id, timeoutMs })` for one bounded run, inspect status, and continue evaluation/integration in the same turn. This is mandatory in headless/one-shot mode; never end that turn while critical children are live. If the probe says `wait` is disabled, use `async: false` from the first launch. Tool calls launch without clarification by default; never set `clarify: true` after autonomous kickoff. Use `context: "fresh"` for competing candidates and adversarial reviewers. Ordinary children must not launch subagents.

## Parallelism and worktrees

- Parallelize independent discovery, read-only review/validation, or intentionally competing candidates only.
- Before each implementation phase, put only the pre-contract integration SHA and ownership/resource plan inside the immutable contract, commit it with an empty mutable evidence seed, then record the resulting full phase contract/candidate-base commit only in mutable run state/evidence. Create durable Git worktrees/branches from that pinned SHA. Before writer launch, atomically create a parent-owned lease outside the worktree containing candidate/phase identity, canonical worktree, resource namespaces, nonce, and launch time; after launch, add the returned child run ID to both the lease and the existing child record in run state. Keep leases separate under ignored/private `.loop/leases/`; do not add an undeclared field to the closed state shape. Give each writer exactly one absolute worktree `cwd` and unique external resource namespace. Do not set Pi-native `worktree: true` for candidates that must survive and merge by SHA.
- Require every candidate to finish with a clean commit, full SHA, exact commands/results, decision assumptions, risks, and no unrelated artifacts.
- Verify each SHA in its own detached evaluation worktree with an absolute `cwd`, fail-fast pre/post `HEAD` assertion, and clean-status assertion. Store reports in ignored `.loop` storage. For one candidate, use an independent exact-SHA `reviewer` and skip comparative bundle construction. For multiple candidates, generate blind patches only from an explicit implementation-path allowlist, normalize reports, scan every regular file for known identity tokens, and reject symlinks/extra files. Give the decisive custom `blind-evaluator` only that sanitized directory—never run state, raw reports, model ledger, or implementer identity. Its read tools accept absolute paths, so a decisive blind claim requires OS mount confinement to the bundle. Reject mandatory-gate failures before scoring. Select one immutable SHA; prefer the smallest safe diff on a true evidence tie.
- One integrator asserts the branch/SHA, audits the effective Git hooks, and merges the selected exact SHA through the clean integration worktree with closed stdin and a hard parent-enforced watchdog timeout. Keep reviewed hooks enabled, capture their output, and mark an interactive/unbounded hook `BLOCKED`. Rerun targeted plus full relevant gates. Never force-delete dirty/unmerged work.

Pi-native `worktree: true` is allowed only for disposable spikes or patch-returning exploration where loss is acceptable. Its patch capture, dependency linking, and cleanup are best-effort; verify retained artifacts.

## Phase and release loop

For each dependency-ready vertical slice:

`SELECT → ISOLATE → TEST → IMPLEMENT → VERIFY → REVIEW → REPAIR/EVALUATE → INTEGRATE → RECORD`

Use deterministic checks before model judgment. Use Sol with actual browser/computer tools for user-facing E2E and state/viewport/accessibility/copy inspection. Run independent final review and integrated regression. Configure OpenWiki only when repository documentation freshness is in scope.

Deploy only to environments explicitly allowed by the authority envelope. Autonomous production requires an exact target, available credentials, health thresholds, progressive rollout, and automatic rollback rules frozen before kickoff. Otherwise stop at the highest authorized environment and record production as not authorized.

Do not finish merely because an agent says done. On restart, reconcile every Pi run/session and parent-owned writer lease before any resume or replacement: verify the owner, canonical worktree branch/HEAD/status, and owned ports/databases/caches/sandboxes/deployment locks. Ambiguous ownership or a lease left at `launching` forbids automatic relaunch and becomes evidence-backed `BLOCKED` if it cannot be resolved safely. `RELEASED` moves run lifecycle to `OBSERVING`; persist a scheduler/monitor ID and next KPI check. This contract terminates only when KPI evidence proves `VALIDATED`/`MISSED_TARGET`, or safe alternatives/global budgets are exhausted and the run is truthfully `BLOCKED`. Before any terminal run seeds a later versioned preflight—including continuation after a blocker is resolved—atomically archive the exact terminal snapshot at `.loop/runs/<run_id>.json`; initialize a distinct `run_id`, link it with `parent_run_id`, and keep old budgets/ledgers immutable in that archive. Never convert “do not stop” into an infinite unchanged retry loop.
